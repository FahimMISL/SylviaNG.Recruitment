# US-070 — Create Multi-Round Interview Configuration + US-072 — View Interviews on a Mobile Interface

## What

HR defines named, sequenced interview rounds per job posting (e.g. "Technical Round 1" then "HR Round"), each optionally paired with a scorecard and suggested panel members. Scheduling an interview can now reference one of these configured rounds instead of a bare typed-in number — doing so auto-derives the `Round` ordinal from the round's sequence, and if it's not the first round, the candidate must already have a `Passed` result on the prior round for that job posting or scheduling is rejected with a clear message. HR marks a completed interview's outcome (Passed/Failed) via a new "Mark Result" action, which also transitions the interview to `Completed` — wiring up a status value that existed but was previously dead. The candidate's HR-facing pipeline tracker shows each interview's round name and result. Separately, the interview list, evaluation, and results pages are usable on a phone: the interview list swaps its table for a stacked card list below 768px, and score-entry/result pages get larger tap targets and a reflowed criteria breakdown.

## Why

Third and final EP-08 (Interview Management) feature, closing out the epic. `Interview.Round` had been a bare, unvalidated `int` since Feature 1, with an entity comment explicitly flagging it as groundwork for this feature. US-072's "mobile access" fits the design decision already made in Feature 2 — HR records evaluations on a panelist's behalf rather than panelists logging in themselves — so "mobile interview dashboard" here means HR's own pages working on a phone, not a new panelist-facing surface.

## Scope decisions

- **`InterviewRoundConfig` is scoped per `JobPostingId`, not per `PipelineStage` or `HiringPipeline`.** AC1 says "for a vacancy" — a job posting can have configured rounds independent of which hiring-pipeline template it uses, matching the existing soft/independent relationship between `Interview` and `PipelineStage`.
- **`Interview.Round` stays a plain int; `InterviewRoundConfigId` is an additive nullable FK.** Job postings with no configured rounds behave exactly as before (free-typed round number, no gating) — this was a hard backward-compatibility requirement since Features 1/2 already have live `Interview` rows with no round config. When a round config is selected at schedule time, `Round` is overwritten server-side from the config's `Sequence` (the frontend disables manual entry once a configured round is picked, but the server is the actual source of truth).
- **Gating (AC3) is enforced only when scheduling against a configured round with `Sequence > 1`.** The check looks up the round at `Sequence - 1` for the same job posting and requires an existing `Interview` for that job application at that round with `Result == Passed` — otherwise `ScheduleAsync`/`BulkScheduleAsync` reject with a message naming the missing round by name. Configured `Sequence == 1` rounds have no gate (nothing to gate on).
- **`Result` is a new field on `Interview` (`InterviewResultEnum: Pending, Passed, Failed`), set only via the new explicit `MarkResultAsync`/"Mark Result" action — not auto-derived from an evaluation's weighted score.** Matches this codebase's precedent of explicit HR-driven state transitions (`Cancel` requires a typed reason, `Reschedule` is a distinct action) rather than an automatic threshold. Marking a result also flips `Status` to `Completed`, finally using the `InterviewStatusEnum.Completed` value the Feature 1 doc flagged as dead code.
- **Round config CRUD is a whole-list replace per job posting** (`PUT recruitment/job-posting/{id}/interview-round-config` takes the full round list), same atomic clear-and-rebuild convention as `HiringPipelineService.UpdateAsync`'s `Stages` collection — `Sequence` is renormalized 1..N from the submitted order server-side, so the frontend's up/down reorder buttons just reshuffle a local array before saving (AC4). Removing a round that's already referenced by an `Interview.InterviewRoundConfigId` is rejected (`ResourceInUseException`) rather than silently orphaning that interview's round link.
- **Round panel members are suggested metadata (`InterviewRoundConfigPanelMember`), not enforced at schedule time.** AC2's "different panel members can be assigned to different rounds" is satisfied by storing them on the config and letting the frontend prefill the panel-IDs field when useful, but the interview's actual `PanelMembers` are still whatever HR types at scheduling — same free-typed-IDs precedent as every other panel field in this codebase, no new employee-picker built.
- **Pipeline tracker (AC5) is the existing HR-facing "Interviews" panel** (`pipeline-progress-tracker.component`, built in Feature 1) — not the candidate-facing `MyApplicationInterviewResponse`, which deliberately omits interviewer feedback/results per its US-040 AC3 comment. Extending that one instead would have leaked pass/fail results to candidates, which was never asked for.
- **Mobile (US-072) reuses the app's existing hand-built responsive idiom** — `isMobile = window.innerWidth < N` plus a `@HostListener('window:resize')`, the same shape as `shell.component.ts`'s sidebar-collapse logic — rather than introducing new breakpoint conventions. `interview-list` is the only page that needed a structural table→card swap (768px breakpoint); this is the first such swap anywhere in the app, since every existing card-grid page (venues, scorecards) never had a table to begin with. `evaluate-interview`/`interview-results` were already reasonably stacked via their existing `sm:` grid classes from Features 1/2, so their mobile work is CSS-only tap-target sizing and a table→stacked-rows reflow for the per-criterion breakdown, not a rebuild.
- **Out of scope, deliberately not built**: a scorecard-driven auto pass/fail threshold, enforcing round panel members onto the actual `Interview.PanelMembers`, blocking round reordering once interviews exist against a posting (AC4 says "before the interview process begins" but this is left as a documented expectation, not a hard runtime guard beyond the delete-when-referenced check), a dedicated panelist mobile login (see Why).
