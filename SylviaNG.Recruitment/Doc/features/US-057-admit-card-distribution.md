# US-057 — Generate and Distribute Admit Cards

## What

Building on the single-candidate admit-card PDF already generated by US-055/US-056, HR can now
bulk-distribute admit cards (email + SMS) to every enrolled candidate for an exam in one action,
bulk-download every candidate's admit card as a single ZIP, and candidates can download their own
admit card from the My Applications portal. The admit card PDF itself now embeds the candidate's
real profile photo when they have one uploaded, and a static invigilator-contact line.

## Why

US-055/US-056's doc explicitly deferred this: "admit-card bulk distribution or candidate-portal
download (US-057)". Continuing the senior-approved EP-07 sequencing (55+56 → 58+59 → 57 → 60 → 61).

## Scope decisions

- **Photo**: `CandidateProfile.ProfilePhotoPath` exists and is already embedded in CV PDFs via
  `QuestPdfCvGenerator.TryLoadPhoto`. `QuestPdfAdmitCardGenerator` now does the same — loads the
  photo file and renders it, falling back to a bordered "No Photo" box when the candidate has none
  uploaded. This corrects an earlier "N/A" stub from the 55/56 build once the photo field was
  discovered to actually exist.
- **Invigilator contact**: no invigilator-identity data model exists — `InvigilatorAssignment` was
  deliberately removed in US-062's `RemoveInvigilatorAssignmentAddRequiredCount` migration;
  `ExamRoom` only tracks a required headcount (`RequiredInvigilatorCount`), not who. The admit card
  renders a static "Contact exam venue administration" line instead of inventing an identity field.
- **Bulk distribution re-runs the same per-enrollment notify path** used at enrollment time
  (`ExamNotificationService.NotifyEnrollmentAsync`, already idempotent and non-throwing) rather than
  a new code path — one exam-wide loop, same email/SMS logic, same status fields.
- **SMS now includes a portal download link** (`{Portal:FrontendBaseUrl}/my-applications`) since
  there's no public/unauthenticated file-serving infrastructure in this codebase (CVs, seat plans,
  and admit cards are all authenticated downloads) — the candidate must log in to get the file,
  same as every other download in this system.
- **Candidate download reuses the same PDF-generation path** (`IAdmitCardPdfGeneratorService`) as
  the HR-facing download, gated by an ownership check (`JobApplication.CandidateProfileId` must
  match the calling candidate), mirroring `ExamTakingService.GetOwnedEnrollmentAsync`'s pattern.
- **Bulk ZIP mirrors `CvBankCvBulkDownloadHandler`'s exact `ZipArchive` pattern** — one PDF per
  enrollment, deduplicated file names.
