# US-066 — Score Interviews Against a Scorecard + US-067 — Configurable Scorecard Templates + US-068 — Aggregate/Export Evaluation Results + US-071 — View Candidate CV During Evaluation

## What

HR builds reusable `Scorecard` templates (a name, description, and a list of weighted criteria — each with a name, weight, and max score) under a new "Scorecard Templates" admin page. For any scheduled interview, HR records an evaluation on behalf of a chosen panelist (picked from that interview's own panel list — no employee picker), selecting one of the active scorecards, entering a raw score per criterion, and an optional overall comment. A `WeightedScore` (`Σ(Score/MaxScore × Weight) / Σ(Weight) × 100`) is computed on read, not persisted. A results page per interview lists every submitted evaluation with per-criterion scores, the weighted total, and an Excel export of the full set. The evaluation form surfaces a link to the candidate's CV so HR can review it side-by-side while entering scores.

## Why

Second of three EP-08 (Interview Management) features, building on Feature 1's `Interview`/`InterviewPanelMember` entities (merged to `demo/local-showcase`). Bundled per senior-approved feature split: US-066/067/068/071 together since they're one coherent data model — a scorecard template must exist before it can be scored against, and the results/export view is just a different read of the same evaluation rows.

## Scope decisions

- **HR records scores on the panelist's behalf, not a panelist self-service login.** `InterviewPanelMember.EmployeeId` is a Kafka-synced org-wide Employee ID with no evidence any such employee logs into this Admin UI — only Admin/HR do. Confirmed with the user before building: evaluation entry is HR picking a panelist from the interview's already-known panel list and typing in that panelist's scores, not a new panelist-identity/auth mechanism.
- **`Scorecard` is an independent, reusable template**, not hung off `PipelineStage` or `Interview`. Investigation confirmed `PipelineStage.MaxMarks`/`PassMarks` is an unrelated "assessment stage" concept wired only into `Exam` scoring — a Scorecard is referenced by `Interview` only at evaluation time (via `InterviewEvaluation.ScorecardId`), the same way `ShortlistFilter` is a standalone reusable template referenced by candidate search, not owned by any one job posting.
- **`ScorecardCriterion` is a homogeneous repeating row** (every criterion has the same `Name`/`Weight`/`MaxScore`/`DisplayOrder` columns) — mirrors `ExamQuestionOption`'s shape, not `ShortlistFilterCriterion`'s type-discriminator shape, since there's no varying criterion "type" to model.
- **Criteria are replaced wholesale on `Scorecard` update** (delete all, re-insert the submitted list) rather than diffed — same atomic-replace convention as `ShortlistFilterService.UpdateAsync` rebuilding its `Criteria` collection.
- **`WeightedScore` is computed in the mapper, never persisted** — same precedent as `ExamEnrollmentMapper`'s derived `AttemptStatus`. Using a ratio-of-sums formula (`Σ(Score/MaxScore × Weight) / Σ(Weight) × 100`) rather than assuming weights sum to exactly 100 makes it robust to any weight scale HR chooses.
- **One evaluation per (Interview, Employee) pair, enforced both by a unique index and a service-level check** (`ExistsByInterviewAndEmployeeAsync` → `DuplicateException` → HTTP 409) — a panelist can't be scored twice for the same interview; HR corrects an existing evaluation via the separate `Update` endpoint instead of resubmitting.
- **Submission validates the employee is actually one of the interview's panel members** (`Interview.PanelMembers.Any(p => p.EmployeeId == request.EmployeeId)`), that the interview isn't `Cancelled`, that the chosen scorecard is `IsActive`, and that the submitted scores cover every one of that scorecard's criteria exactly once with each score in `[0, MaxScore]` — no partial-criteria submissions.
- **Candidate CV link reuses the existing `JobApplicationDetail.resumeUrl`** (already available via `JobApplicationService.getDetail`, same "open resume in new tab" pattern as `application-detail.component.html`) — no new backend endpoint or file-storage work needed for US-071.
- **Frontend criteria editor is a plain array with push/splice + `*ngFor`**, not `FormArray` — there is no `FormArray` precedent anywhere in this codebase; matches the dominant plain-array-editing style used elsewhere (e.g. shortlist-filter criteria rows).
- **Out of scope, deliberately not built**: multi-round scorecard gating/sequencing (Feature 3, US-070), a mobile-optimized evaluation view (Feature 3, US-072), an employee-lookup picker for panelists (same deferred-picker precedent as US-062/US-063).
