# US-063 — Schedule Interview + US-064 — Reschedule/Cancel Interview + US-065 — Conflict Detection + US-069 — Manage Interview Venues and Rooms

## What

HR schedules an `Interview` (in-person or virtual) for a shortlisted candidate against a job posting, either one at a time or in bulk for several candidates at once — bulk-scheduling auto-staggers each candidate into a back-to-back slot from a shared start time, by the given duration + gap. In-person interviews are tied to an `InterviewVenue`/`InterviewRoom` (a standalone registry HR manages separately, mirroring US-062's exam venue/room pattern); virtual interviews just need a meeting link. A panel of interviewers is captured as free-typed Employee IDs (no picker — same precedent as US-062 and the hiring-pipeline's `interviewerEmployeeIds`), modeled as a real `InterviewPanelMember` join table. Scheduling (single or bulk) is rejected if the room is already at capacity for that time slot, or if any panelist already has an overlapping interview elsewhere. HR can reschedule (new date/time, optionally a new venue/room or meeting link) or cancel (with a required reason) any interview, individually or in bulk; each action emails the candidate (real SMTP) and logs an SMS (stub), same non-blocking notification pattern as US-055/056. The candidate's pipeline-progress tracker shows a flat list of their scheduled interviews with a link to reschedule/cancel.

## Why

First of three EP-08 (Interview Management) features, approved to build in this order because Feature 2 (Evaluation & Scorecards) and Feature 3 (Multi-Round Config) both need a real `Interview` entity to attach to. Bundled per senior-approved feature split: US-063/064/065/069 together since they're one coherent scheduling data model (an `Interview` needs a venue/room to exist before it can reference one, and the reschedule/cancel/conflict-check logic is all one service).

## Scope decisions

- **Replaced the pre-existing `Interview` entity rather than extending it.** A table named `Interviews` has existed since 2026-07-07 (`AddJobVacancyConfigurationFields` migration) with a single `InterviewerId`, no venue, no status enum — but it was never wired to any controller/service/command in 16 days of feature work. One live consumer *was* found during investigation: `JobApplication.ToMyApplicationResponse()`/`MyApplicationInterviewResponse` (US-040 "My Applications" candidate view) reads `Interview.ScheduledDate/Location/MeetingLink/Round/IsActive` — updated to the new shape (`ScheduledStartAt`, derived venue/room-or-meeting-link string, `Round` as int, `Status != Cancelled` instead of `IsActive`). Since nothing ever wrote to the old table, this candidate-facing list was always empty in practice; replacing the schema doesn't change that.
- **Panel members**: free-text comma-separated Employee IDs on the frontend (no lookup/picker — matches the deferred-picker decision already made for US-062 invigilators), but modeled as a genuine `InterviewPanelMember` join table server-side (composite key + FK to `Employees`, cascade delete), same shape as `PipelineStageInterviewer`.
- **Separate `InterviewVenue`/`InterviewRoom` entities, not reused `ExamVenue`/`ExamRoom`.** Those are FK'd specifically into the Exam domain; there's no shared/polymorphic Venue table in this codebase, so a parallel pair was the right call rather than forcing cross-domain reuse.
- **Room capacity = max concurrent interviews in that room**, checked as a time-overlap count (`existing.Start < newEnd && existing.End > newStart`) against non-cancelled interviews — new logic, since nothing in this codebase previously checked time-range overlap (Exam's capacity check is purely count-based, one exam = one fixed time for everyone).
- **Panelist double-booking check** is the same time-overlap idea, joined through `InterviewPanelMember.EmployeeId` across all interviews, independent of room/type (applies to virtual interviews too).
- **Venue is derived from the room, not taken from the caller** at schedule time (`ResolveLocationAsync`) — same "trust the child, ignore a possibly-stale parent id" precedent as `ExamEnrollmentService.ReassignSeatAsync`'s venue-consistency check.
- **`PipelineStageId` is a soft reference** (nullable, no FK), same convention as `JobApplicationStageProgress.PipelineStageId` — pipeline templates can be edited independently of scheduled interviews. Not exposed in the schedule-interview form for this feature (no stage-lookup-by-job-posting UI was built — out of scope, the field just isn't set yet); the pipeline tracker hook shows all of a candidate's interviews as one flat list rather than attached to a specific stage card, for the same reason.
- **Bulk-schedule staggers by `DurationMinutes + GapMinutes`** starting from a shared `StartAt` — mirrors `ExamSeatPlanService.GenerateAsync`'s "compute then bulk-persist" shape, but time-based instead of count-based. Bulk-reschedule staggers similarly but keeps each interview's own existing duration/venue/room, only shifting time.
- **No enforced state-machine beyond terminal-state guards** — `Reschedule`/`Cancel` just check the interview isn't already `Cancelled`; no `Completed`/`NoShow` transition logic is wired yet (those enum values exist for a future feature to set).
- **Out of scope, deliberately not built**: employee-lookup endpoint/picker (per US-062 precedent), a `Round`-sequencing/gating feature (Feature 3, US-070), scorecards/evaluation (Feature 2), admit-card-style PDF generation for interviews.
